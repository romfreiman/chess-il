# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Data Pipeline. Generated by gsd-ui-researcher.

---

## Phase Classification: Backend Only

Phase 1 delivers the scraping pipeline, Supabase caching layer, and REST API. It contains **zero frontend components, zero UI screens, and zero user-facing visuals**. All requirements (SCRP-01..05, CACH-01..05, API-01..03) are backend-only.

The first UI surface appears in Phase 2 (Home & App Shell). This spec documents:
1. Project-wide design tokens locked in CLAUDE.md and REQUIREMENTS.md, establishing the foundation for Phases 2-5
2. API response copywriting (error messages returned as JSON, consumed by frontend in later phases)

## Copywriting Contract

Phase 1 produces API error response messages as JSON. These are the source of truth for frontend error display in Phases 2-4.

| Element | Copy |
|---------|------|
| Primary CTA | not applicable (no UI in this phase) |
| Empty state heading | not applicable (no UI in this phase) |
| Empty state body | not applicable (no UI in this phase) |
| API error: invalid ID | `{ error: "INVALID_ID", message: "Player ID must be a positive integer", statusCode: 400 }` |
| API error: player not found | `{ error: "PLAYER_NOT_FOUND", message: "No player found with ID {id}", statusCode: 404 }` |
| API error: scrape failed (no cache) | `{ error: "SCRAPE_FAILED", message: "Unable to fetch player data. Please try again later.", statusCode: 502 }` |
| API meta: stale data flag | `meta.stale: true` (frontend in Phase 4 will display appropriate Hebrew warning) |
| Destructive confirmation | not applicable (no destructive UI actions in this phase) |

Source: CONTEXT.md decisions D-05 and D-06 (locked API response shape)
